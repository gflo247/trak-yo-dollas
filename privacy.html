💾 Your data saves automatically

The app saves everything — accounts, balances, imported transactions, budgets, and snapshots — to your browser's local storage. It persists across refreshes and restarts, even without signing in. Your browser processes CSV files locally and nothing leaves your device unless you choose to sync. You can export your transactions and budget history as a CSV anytime from the Spending and Budget tabs, respectively.

🔁 Sign in to sync across devices

Sign in with Google and the app backs up your full data — accounts, transactions, budgets, categories, and snapshots — directly to your Google account via Firebase. When you sign in, your transaction data leaves your device — Google stores it in your personal Google account on their servers. Google enforces access at the database level so only you can read it — not the developer. Sign in on any device to restore everything instantly. If you'd rather your data never leave your device, skip sign-in — the app works fully without it.

🗂 Community categorization rules

Each time you open the import screen, the app downloads community-rules.json from the app's Firebase hosting once per session. This file contains keyword→category mappings the app uses to auto-categorize your transactions. The app sends no transaction data, descriptions, or personal information — it only downloads the rules file. The file lives publicly on GitHub so anyone can audit it. Missing a merchant? Suggest a pattern ↗ — no account required. You can block this download using any privacy extension — the app will still work, but will skip applying community categorization rules to your transactions. Your own custom rules still apply.

Two separate types of data collection happen when you use this app — they're easy to confuse:

  • Server logs — automatic records that any web server keeps when it delivers files to your browser. Firebase records your internet address and browser type when it sends the app to your device (this is standard for any website you visit). This happens on every visit and can't be turned off — it's how the web works.
  • Analytics — privacy-respecting page view counts (no cookies, no personal data, no cross-site tracking). This is a separate system from Firebase. You can block it with any privacy extension.
📊 Analytics — what's collected

This site uses Umami, a privacy-respecting analytics tool. It records page views, referrer, browser type, operating system, and country (inferred from your IP address, which Umami never stores). Umami sets no cookies, collects no personal data, and never identifies or tracks individual visitors across sessions. You can block it with any privacy extension — the app works fine without it.

🖥 Firebase hosting logs

When you visit this app, Firebase Hosting — Google's infrastructure — records standard server request data: your internet address, browser type, pages requested, and timestamps (this is standard for any website you visit). Google manages this logging under their own privacy policy — the developer has no control over it and no direct access to individual request logs. It contains no financial data; Google applies the same logging to every website on their infrastructure. See firebase.google.com/support/privacy for details.

Frequently asked questions

Can the developer see my financial data?+

No — and this isn't just a policy. It's how the app is built. By default, the app saves your data to your browser on your device. The developer built no mechanism to transmit it anywhere and runs no server to receive it. If you choose to sign in with Google, your data syncs to your personal Google account via Firebase — Google holds it, not us, and only you can read it. Either way, the developer has no way to see your financial data.

Could a hacker access my financial data?+

No — and this isn't just a policy, it's structural. Neither the app nor the developer holds your financial data, so there's nothing to steal. That said, here are the three things someone could theoretically attack, and how each is protected:

  • The app files — hosted on Firebase. Deploying any change requires the developer's Google account credentials plus their physical machine, and that Google account uses two-factor authentication (a password plus a second confirmation, like a phone prompt). An attacker would need both to make any change.
  • The source code — stored on GitHub, also protected with two-factor authentication. Because the source is public, anyone can verify exactly what the app does and spot anything unexpected.
  • Your synced data — only relevant if you've signed in with Google. Firebase security rules enforce that your data is accessible only with your own Google credentials. An attacker would need to compromise your personal Google account, not the developer's.

None of these paths expose your financial data — the app and the developer simply don't hold it.

What if this app shuts down?+

Two scenarios:

  • The website stops working — you'd see a 'page not found' message when you try to open it, but your data is unaffected. Without signing in, it lives on your device. With Google sign-in, it lives in your personal Google account — an account you own directly with Google, not through this app.
  • The developer stops updating it — the app keeps working exactly as-is. No data loss, no changes.

In either case, export your transactions from the Spending tab and budget history from the Budget tab anytime. Your data outlives the app.

Is my CSV safe after I import it?+

Yes — and not just because I say so. Your browser processes the CSV entirely on your device; nothing is uploaded. The app reads the file, parses the transactions locally, and discards the rest — the developer never receives the file or its contents. If you've signed in with Google, the parsed transaction records sync to your personal Google account, but your original CSV file never leaves your computer.

Could the developer share my data with governments or law enforcement?+

No — and this is technical, not just policy. If you haven't signed in, the app stores your data only on your device and the developer receives none of it, so there's nothing to hand over. If you have signed in with Google, your data lives in your personal Google account — any legal request would need to go to Google directly, not to the developer of this app. Either way, there's nothing for the developer to hand over.

Does this app use cookies?+

No — Umami, the analytics tool this site uses, sets no cookies and uses no persistent identifiers. Google's infrastructure may set cookies if you choose to sign in with Google for the optional sync feature. You can block analytics entirely with any privacy extension.

What if my device has spyware on it?+

Spyware is malicious software that secretly records what happens on your device — it could capture anything visible on your screen or stored locally, including data this app saves to your device. This risk applies to all software that runs locally, not just this app. The developer can only control what's in the app's code — your device is yours to protect. Keeping your operating system and browser updated is the single most effective step you can take.

Does the app work without an internet connection?+

Yes — after the initial page load, the app runs entirely in your browser with no connection required. You can view your data, add transactions, and use every feature offline. The only things that need a connection are: loading the app for the first time, downloading the community rules file when you open the import screen (optional — the app works without it), and syncing if you're signed in with Google.

What is Firebase and why use it if you care about privacy?+

Honest answer: Firebase is Google infrastructure. It's used here for two things only — static file hosting (delivering the app's HTML, JS, and CSS to your browser, the same role any web host plays) and optional Google account sync. I chose it for reliability and because it offers a free tier suitable for an indie project. The tradeoff is that Google's servers handle file delivery and log standard request data. If that's a concern, the source code is public and the app can be self-hosted.

Can I self-host this?+

Yes. The full source code is on GitHub — it's a single HTML file with no build step or server required. Download it and open it directly in your browser, or host it on any static file host. Your data stays wherever you run it.

What third-party code runs in the app?+

The app uses D3.js, a widely used open-source charting library, to draw the spending visualizations. Like most web apps, it loads this library from jsDelivr — a content delivery network (CDN) that serves open-source software files. Your browser fetches this file when the app loads, the same way it fetches any webpage resource. jsDelivr receives no financial data — it only delivers a JavaScript file. The same applies to the Firebase software development kit (SDK), loaded from Google's CDN. If you block either request using a privacy extension, your financial data remains completely safe — charts won't render without D3, but all your data stays intact and accessible.

Can other websites read my data?+

No — and this isn't just a policy, it's how browsers work. Each website gets its own private storage, and your browser strictly enforces those boundaries. No other site can reach what this app stores — not by policy, but by technical design.
